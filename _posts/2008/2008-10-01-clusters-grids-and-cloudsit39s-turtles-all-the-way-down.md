<p>
   In this blog, I'll discuss open problems and new developments in the field of distributed systems.
  </p>
  <p>
   A distributed system is a set of independent computers that accomplish a meaningful task by working together. The field covers a huge range of interesting systems, from sensor networks to multi-tier web server farms. For the most part, I will be writing about the sort of large computing systems used to attack very large problems generated by big science and big business.

These systems go by many different names that mean very similar things:

  </p>
  <ul>
   <li>
    A
    <strong>
     cluster
    </strong>
    is a distributed system that consists of a number of identical machines owned by a single entity, usually stacked up in a closet or a machine room. Clusters became the most common form of high performance computing in the 1990s and are the type of system now dominating the
    <a href="http://www.top500.org/">
     Top 500 List
    </a>
    of supercomputers.
   </li>
   <li>
    A
    <strong>
     grid
    </strong>
    is a distributed system that enables people to access computing resources from different institutions over the wide area. The term grid computing was coined by
    <a href="http://ianfoster.typepad.com/">
     Ian Foster
    </a>
    and Carl Kesselman in the late 1990s to describe easy access to large scale computational power. Examples of grids include
    <a href="http://www.teragrid.org/">
     TeraGrid
    </a>
    and the
    <a href="http://www.opensciencegrid.org/">
     Open Science Grid
    </a>
    .
   </li>
   <li>
    A
    <strong>
     cloud
    </strong>
    is a distributed system where the user doesn't care exactly what resources are used to carry out an operation; this is virtualization in the most abstract sense. There exist commercial clouds such as
    <a href="http://aws.amazon.com/ec2/">
     Amazon EC2
    </a>
    , as well proprietary clouds found in nearly any industrial scale web site.
   </li>
  </ul>
  <p>
   Although people love to argue about these terms, the differences aren't terribly important. In fact, the terms often represent different facets of the same systems. When you submit a job to a
   <strong>
    cluster
   </strong>
   , you are treating it like a
   <strong>
    cloud
   </strong>
   , because you don't care on exactly which CPU the job runs. A
   <strong>
    grid
   </strong>
   is usually built up from multiple
   <strong>
    clusters
   </strong>
   and connected together by wide area networks and software. A
   <strong>
    cloud
   </strong>
   is usually contained in a single data center. If you need to access it remotely, then you need a
   <strong>
    grid
   </strong>
   .
  </p>
  <p>
   If you want to really want to mix it up, read about Ed Walker's
   <a href="http://www.tacc.utexas.edu/~ewalker/mycluster.html">
    MyCluster
   </a>
   , which is a system that uses a
   <strong>
    grid
   </strong>
   to allocate a personal
   <strong>
    cluster
   </strong>
   , which you can then use as a
   <strong>
    cloud!
   </strong>
  </p>
  <p>
   Regardless of what we call these systems, the challenges are the same. How do we design the interactions between pieces so that the system is robust to failures, has acceptable performance, and protects the interests of all of the parties involved? Future posts in this column will focus on these fundamental problems.
  </p>
  <p>
   So that you know where I am coming from, I am a professor at the
   <a href="http://www.nd.edu/">
    University of Notre Dame
   </a>
   , where I teach classes in distributed systems, operating systems, and compilers. I direct the
   <a>
   </a>
   <a href="http://www.cse.nd.edu/~ccl">
    Cooperative Computing Lab
   </a>
   where a
   <a href="http://www.cse.nd.edu/~ccl/research/people.shtml">
    great group of students
   </a>
   does the hard work to design and test new distributed systems. We develop solutions to problems in physics, chemistry, biometrics, and other fields, and them evaluate them on our shared testbed of 700 CPUs. I'll be writing more about these ideas in this column.
  </p>
